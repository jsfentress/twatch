# TUI Shell

| Component | TUI Shell |
| --- | --- |
| Purpose | The Textual `App` that renders a two-pane view (session tree on the left, details on the right), polls tmux on a background worker, and brokers all interactive actions (new, attach, refresh). It is the user-facing surface of twatch; correctness here means the screen reflects current tmux state within one poll interval, attaching to a session never corrupts the terminal, and store mutations are flushed atomically. |

| Best Practices | **The store is loaded once in `__init__` and persisted only at decision points** — `on_mount` and `action_new_session`. Reactive watchers do not save. Keep it that way; `save_store` is atomic but unnecessary churn is still churn. **`poll_tmux_loop` runs as a Textual worker via `run_worker(..., exclusive=True, group="poll")`** — do not start a second poller and do not switch to a synchronous timer; `asyncio.to_thread` keeps the tmux call off the event loop. **`sessions` is `reactive(list, always_update=True)`** so reassigning the same list still fires `watch_sessions`; do not mutate it in place. **Attaching is dual-mode** — inside tmux, `tmux.attach` returns a `CompletedProcess` and the app exits cleanly; outside tmux, `self.suspend()` is mandatory or Textual and tmux fight over the terminal. After the suspend block, always call `self.refresh_now()` so the tree reflects the post-attach state. **Use `self.notify()` for transient feedback**, not a custom status line — Textual already gives you toast notifications with severity levels. **Surface tmux stderr via `notify(..., severity="error")`** truncated to ~120 chars; do not raise. **`_rebuild_tree` re-selects via `call_after_refresh`** because the cursor line isn't valid until Textual lays out the tree. **Sorting is by display title (lowercased), not raw session name** — that's what users see, so it's what should determine row order. |
