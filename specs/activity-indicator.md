# Activity Indicator

| Component | Activity Indicator |
| --- | --- |
| Status | **Not yet implemented in the Textual port.** This spec describes the intended design and the data already in place to support it. Treat this file as a forward-looking contract: any code added under this name must match it. |
| Purpose | Show which tmux sessions have had recent output by visually marking their row in the sidebar tree for a short window after a poll detects new activity. Gives the user a passive, always-on signal of "something just happened over there" without having to attach. The indicator must be responsive enough to feel live (sub-second after a poll) but stable enough not to flicker on every redraw. Selected and attached states must remain visually distinguishable from "recently active." |
| File map with roles | `twatch/tmux.py:10` `FMT` — already includes `#{session_activity}`, tmux's monotonically-increasing per-session activity counter. `twatch/tmux.py:38-61` `list_sessions` — already populates `"activity": int(activity)` on every session dict; the value is fetched but not yet consumed by the UI. `twatch/app.py:20` `POLL_SECONDS = 0.5` — the poll cadence the indicator must keep up with. `twatch/app.py:257-265` `poll_tmux_loop` — the natural place to compute the per-tick diff: snapshot the previous `activity` map, compare against the fresh one, and update a `{name: monotonic_deadline}` map before reassigning `self.sessions`. `twatch/app.py:140-173` `_rebuild_tree` — the natural place to consume the deadline map: prepend a `[bold yellow]●[/bold yellow]` (or similar) marker to active rows alongside the existing attached/dead markers, or apply a CSS class via Textual's reactive system. `twatch/app.tcss` — where any CSS class for "active" should live so the look stays consistent with the rest of the UI. |
| Best Practices | **Use `time.monotonic()`, not `time.time()`** — the active-window deadline must be immune to wall-clock jumps (NTP, suspend/resume). **Seed the previous-activity snapshot from the first `list_sessions` call** in `on_mount` so existing sessions don't all flash active on TUI startup. Do not reset it mid-loop. **Tmux `session_activity` is a counter, not a boolean** — compare strictly greater, not equal; equal means "no new output this tick." **Clean up the snapshot and deadline maps for dead sessions** or you leak memory and, worse, a recreated session with the same name inherits stale state. **Do not derive activity from `attached`** — a user may be attached and idle, or detached and actively producing output; only `session_activity` is meaningful. **Prefer Textual's CSS/reactive system over inline markup** for the active state — define a class in `app.tcss` and toggle it on the tree node, so the visual is themeable in one place. If you must use inline markup, keep it consistent with the existing `[green]●[/green]` / `[red]✗[/red]` glyph vocabulary. **The selected row and the active state must remain distinguishable** — Textual already inverts the cursor row, so an active marker should be a glyph or color that survives the inversion. Test both with and without the cursor on the active row. **The active window should be ~1–2 seconds**, long enough to register visually at a 0.5s poll cadence but short enough that activity feels live. Define the constant in `app.py` next to `POLL_SECONDS`. |
