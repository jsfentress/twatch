# Tmux Bridge

| Component | Tmux Bridge |
| --- | --- |
| Purpose | The single boundary between twatch and the `tmux` binary. Every read of live session state and every mutation (create, attach, kill) goes through here. Isolating tmux calls in one place means the rest of the code never has to think about `subprocess`, tmux format strings, or the difference between "tmux missing" and "tmux ran and returned nothing." |
| File map with roles | `twatch.py:15` `FMT` — the `list-sessions` format string (`name\|attached\|activity`). `twatch.py:22-23` `tmux_ok` — PATH probe, used by CLI entry points before doing anything. `twatch.py:26-46` `list_sessions` — parses `tmux list-sessions -F FMT`; returns `None` if tmux is missing, `[]` if no sessions, else list of `{name, attached, activity}`. `twatch.py:101-103` `attach` — ends curses, execs `tmux attach -t`. `twatch.py:263-282` `new_session` (TUI) and `twatch.py:461-481` `cmd_create` (CLI) — both call `tmux new-session -d -s NAME [cmd]`. `twatch.py:285-291` `kill_session` — `tmux kill-session -t NAME`. `twatch.py:430-458` `cmd_register` — uses `tmux display-message -p '#{session_name}'` to resolve current session from `$TMUX`. |
| Best Practices | **`list_sessions` returns `None` vs `[]` deliberately** — `None` means tmux is unreachable (binary missing, fork failed), `[]` means tmux ran fine and there are simply no sessions. Callers must distinguish these; the TUI renders "tmux not found" for `None` and "no sessions" for `[]`. **Always call `curses.endwin()` before `tmux attach`** — otherwise curses and tmux fight over the terminal. See `attach` at line 102. **Never shell out with `shell=True`** — all tmux invocations pass an argv list so session names and commands can't be shell-injected. **Keep the `FMT` pipe-delimited with a `split("|", 2)`** — session names cannot contain `|` in tmux, but activity/attached are numeric, so a bounded split is safe and avoids regex. **Register resolves the current session via `tmux display-message`, not `$TMUX`** — `$TMUX` only tells you that you're *inside* tmux, not which session; display-message is the canonical way. |
